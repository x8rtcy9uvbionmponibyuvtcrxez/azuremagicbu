The loophole:

ThereГўв‚¬в„ўs a big misconception that these inboxes are special in one way or another.

Words like Гўв‚¬Е“enterpriseГўв‚¬Вќ or Гўв‚¬Е“azureГўв‚¬Вќ get thrown around.

But theyГўв‚¬в„ўre simply just Outlook shared mailboxes.

Shared mailboxes are essentially standard outlook mailboxes, just with limited functionality. Meaning theyГўв‚¬в„ўre only purpose is to send and receive emails.

TheyГўв‚¬в„ўre primary use case is for collaborative accounts like info@ or support@ inboxes, but the key loophole is you can actually assign passwords to these mailboxes which allows you to connect them to your sender to send and receive emails.

And most importantly, they DON'T need a paid license/subscription to send and receive emails.

So you only need to pay $4.50/m for 1 Exchange Online license -> assign it to 1 regular user mailbox and use the other 99 inboxes for Гўв‚¬Е“freeГўв‚¬Вќ.

But the tricky part, is being able to setup working infrastructure without running into major issues or getting flagged with Microsoft.

The system is essentially a puzzle, where every piece is crucial and without one - the whole system breaks.

Note on Гўв‚¬Е“Azure inboxesГўв‚¬Вќ:

These are NOT Гўв‚¬ЛњAzure InboxesГўв‚¬в„ў. ThereГўв‚¬в„ўs technically no such thing as Azure Inboxes.

Azure is simply a hosting provider owned by Microsoft.

What you would consider azure inboxes are rare, and are actually just private SMTP servers hosted on azure with a custom email server hosted on it.

ThatГўв‚¬в„ўs an entirely different system to this one.

Deliverability

Deliverability in terms of infrastructure (not including email copy or other little factors) comes down to 2 things:

	1	Domain health/reputation
	2	Server health/reputation

Health or reputation essentially means - how do other major ESPs such as Google or Microsoft view my domain or server? Do they recognise it as spam or safe?

Good health = Google, Microsoft etc. donГўв‚¬в„ўt view your domain or server as spam.

And vice versa

So when it comes to these inboxes (and Google) your only consideration is domain health.

Because Google and Microsoft have such a major infrastructure thatГўв‚¬в„ўs used by majority of the population, that their servers are essentially Гўв‚¬Е“bulletproofГўв‚¬Вќ and cannot have a bad reputation.

Therefore if deliverability is poor, and all other factors are considered - itГўв‚¬в„ўs likely your domain health.

Limits & Considerations

As you already know, Microsoft does not like people misusing their platform. Which is why you need to take measures to ensure you donГўв‚¬в„ўt get flagged.

There are limits.

	1	Tenant sending threshold:
	1	Every new tenant has a Гўв‚¬Е“hidden sending limitГўв‚¬Вќ which is around 2,000 emails/day (across ALL inboxes). This includes cold emails, warmup emails and warmup replies.
	2	This is the reasoning behind the warmup and sending settings we recommend.
	2	Never have inboxes warming up on more than 1 platform at a time.
	1	For example if you had inboxes connected to instantly and emailbison, with warmup enabled in both that would be a big NO NO.
	3	New tenants have a 3 day Гўв‚¬Е“warmup periodГўв‚¬Вќ.
	1	Another hidden limit from Microsoft. Within three days of setting up a tenant, if you attempt to connect inboxes to a sender -> and send an email, youГўв‚¬в„ўll likely hit an IP error.
	2	You simply just need to wait out the 3 days and it will resolve itself.

Risk Mitigation (IMPORTANT)

	1	Random licenses:
For every 10 - 20 tenants you, set up 1 - 2 with random licenses other than just Exchange Online Plan 1.

These can be any licenses, but the point is to show youГўв‚¬в„ўre not just solely subscribing to the cheapest product that allows you to send emails - this screams spam.

	2	Spread across tenant providers:

All of your tenants should be spread across at least 2 tenant providers. For example if you need 100 tenants - setup 50 on Intermedia and 50 on Sherweb.

Cloudflare Setup:

Cloudflare will be the platform where you host all of your domains, to allow you to easily manage all of the DNS records and redirects automatically.

What it does:

Cloudflare hosts all the DNS records for your domains, including MX records for email delivery, SPF/DKIM/DMARC records for email authentication, and A records for website forwarding.

The automation scripts communicate directly with Cloudflare's API to automatically configure all these DNS settings when you add a new domain.

Why you need it:

Manually configuring DNS records for every domain would take hours. Cloudflare's API allows the scripts to automatically create all necessary DNS records in seconds. Additionally, Cloudflare provides website forwarding so your domains can redirect to any URL you choose.

RDP Setup:

RDPs (Remote Desktop Protocol) are virtual Windows computers hosted in data centers that you can connect to remotely from your own device. Think of it as renting a computer in the cloud that runs 24/7.

What it does:

An RDP provides a clean, dedicated Windows environment where you can run all the automation scripts without affecting your personal computer. It acts as the "command center" where all domain and inbox creation happens.

Why you need it:

Microsoft monitors login patterns and device fingerprints. If you create dozens of Microsoft 365 tenants from your personal computer or home network, Microsoft will flag your activity as suspicious and potentially ban your accounts.

RDPs provide a consistent, US-based IP address and clean device
fingerprint that keeps Microsoft from detecting automation patterns. Additionally, RDPs allow the scripts to run continuously without keeping your personal computer on.

RDPs can be rented on a subscription from a bunch of hosting providers, but we recommend ContaboГ‚В as they give you the best bang for your buck.

RDP Setup (Powershell):

Powershell is the main application that makes this system possible. It manages the entire domain and inbox setup process.

What it does:

PowerShell runs all the automation scripts that create Microsoft 365 tenants, add domains, create users, set up mailboxes, configure passwords, and manage permissions.

It connects directly to Microsoft's cloud services through special modules (ExchangeOnlineManagement and Microsoft.Graph) that give it the ability to perform administrative tasks programmatically.

Why you need it:

Creating 99 inboxes manually through Microsoft's web interface would take 6-8 hours per domain and be extremely error-prone. PowerShell scripts automate this entire process down to 30 minutes per domain with zero manual clicking.

How it works:

We provide you with pre-written PowerShell scripts (files ending in .ps1) that contain sequences of commands. When you run a script, PowerShell executes each command in order: connecting to Microsoft 365, authenticating with your Azure App credentials, creating mailboxes, setting passwords, updating DNS records through Python helper scripts, and tracking progress in Airtable.

The scripts handle all the complex API calls, error handling, and retry logic automatically. You simply run the script, provide the domain name when prompted, and PowerShell does all the work.
